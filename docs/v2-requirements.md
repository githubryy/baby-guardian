# 宝宝守护者 V2.0 需求规划文档

> 基于 V1.0 已有架构（Vue 3 + uni-app + uView UI + 微信云开发），围绕「家庭协作」「成长记录」「智能提醒」「数据导出」「体验优化」五大方向规划二期需求。

---

## 一、现状分析（V1.0 已有能力）

| 能力 | 说明 |
|------|------|
| 提醒事项 | 6 种类型（喂养/换尿布/哄睡/维生素D/用药/自定义）+ 自定义间隔 |
| 推送机制 | 一次性订阅消息 + 配额管理（7 天时效）+ 红点降级 |
| 统计 | 周/月完成率趋势 + 事项类型分布 + 概览数据 |
| 数据模型 | 7 个集合（users/babies/reminder_tasks/confirm_logs/subscription_quotas/notification_logs/families） |
| 预留 | families 表已建但空置，User 表已有 familyId 字段 |

---

## 二、V2.0 五大需求方向

### 方向一：家庭协作（核心，优先级最高）

V1.0 是纯单用户模式，一个宝宝只能归属一个 openId。育儿场景中父母双方、祖辈、月嫂都需要协同，这是 V2.0 最核心的诉求。

#### 2.1.1 家庭组管理

| 功能 | 说明 |
|------|------|
| 创建家庭 | 用户主动创建家庭组，生成邀请码/小程序码 |
| 邀请成员 | 分享小程序码或微信邀请链接，扫码即加入 |
| 成员角色 | owner（创建者）、admin（管理员）、member（普通成员） |
| 移除成员 | owner/admin 可移除成员；成员可主动退出 |
| 家庭设置 | 家庭名称、默认提醒接收人、操作通知开关 |

#### 2.1.2 宝宝共享

| 功能 | 说明 |
|------|------|
| 宝宝归属变更 | 宝宝从「归属于单个 userId」改为「归属于 familyId」 |
| 数据迁移脚本 | V1.0 升级时，自动将每个用户的 familyId 设为自身 ID，宝宝归属迁移到家庭 |
| 多成员可见 | 家庭内所有成员都能看到共享宝宝的提醒事项和历史记录 |
| 指定负责人 | 每个事项可指定主要负责人，避免重复操作 |

#### 2.1.3 协同操作

| 功能 | 说明 |
|------|------|
| 实时同步 | 家庭成员操作（完成/延迟/添加事项）实时同步，使用微信云数据库 `watch` 实时推送 |
| 操作记录 | 记录每条操作由谁执行（confirm_logs 新增 operatorId + operatorName 字段） |
| 防重复完成 | 乐观锁机制，一个事项被 A 完成后 B 端立即标记为已完成 |
| 操作通知 | 家庭成员完成事项时，推送轻量通知给其他成员（“爸爸已完成 15:00 喂奶”） |

#### 2.1.4 数据模型变更

```
families 集合（激活）:
  + name: 家庭名称
  + inviteCode: 邀请码（6位）
  + ownerUserId: 创建者
  + members[]: [{ userId, role, joinedAt, nickName, avatarUrl }]
  + settings: { notifyOnComplete, defaultAssignee }
  + createdAt

babies 集合:
  - userId → familyId（归属改为家庭）
  + createdBy: 创建者 userId

reminder_tasks 集合:
  + assigneeId: 指定负责人 userId（可选）
  + familyId: 家庭 ID（冗余字段加速查询）

confirm_logs 集合:
  + operatorId: 操作人 userId
  + operatorName: 操作人昵称
  + operatorAvatar: 操作人头像
```

#### 2.1.5 新增云函数

| 云函数 | 说明 |
|--------|------|
| `api-family` | 家庭组 CRUD + 邀请码生成/验证 + 成员管理 |

### 方向二：成长记录

V1.0 只记录“是否完成提醒”，缺少量化和纵向的成长数据。育儿场景中身高体重曲线、喂养量、发育里程碑是家长非常关心的内容。

#### 2.2.1 身高体重记录

| 功能 | 说明 |
|------|------|
| 录入 | 手动录入身高(cm)、体重(kg)、头围(cm)、测量时间 |
| 成长曲线 | WHO 标准生长曲线图（Z-Score 3rd/15th/50th/85th/97th 百分位） |
| 对比 | 当前测量值落在哪个百分位区间，自动标注偏瘦/正常/偏胖 |
| 趋势 | 折线图展示历史趋势，支持按月/季/年查看 |

#### 2.2.2 喂养量记录

| 功能 | 说明 |
|------|------|
| 奶量记录 | 每次喂养记录奶量(ml)，关联到喂养类型事项的确认操作 |
| 母乳/配方奶 | 区分母乳、配方奶、混合喂养，记录左/右侧时长 |
| 辅食日记 | 6 个月+ 宝宝的辅食添加记录（食材、量、过敏反应） |
| 每日汇总 | 每日总奶量、喂养次数、辅食种类统计 |

#### 2.2.3 发育里程碑

| 功能 | 说明 |
|------|------|
| 预设里程碑 | 按月龄推送预设里程碑（抬头/翻身/坐/爬/站/走/说话等） |
| 自定义记录 | 家长可手动记录宝宝“第一次”时刻（第一次笑、第一次叫妈妈） |
| 月龄对照 | 当前月龄 vs WHO 标准发育时间表，标注已达成/待观察 |
| 提醒关联 | 到达里程碑月龄时自动创建提醒事项 |

#### 2.2.4 新增数据集合

| 集合 | 关键字段 |
|------|---------|
| `growth_records` | babyId, type(height/weight/headCirc), value, measureDate, note |
| `feeding_details` | babyId, taskId(关联确认), feedType(breast/formula/mixed), amount, side, duration, foodName(辅食) |
| `milestones` | babyId, milestoneKey, expectedAge, achievedDate, note, isCustom |

#### 2.2.5 新增云函数

| 云函数 | 说明 |
|--------|------|
| `api-growth` | 成长数据 CRUD + 百分位计算 |
| `api-milestone` | 里程碑 CRUD + 月龄自动匹配 |

### 方向三：智能提醒

V1.0 的提醒间隔是用户手动设置固定值。实际育儿中，宝宝的作息会随月龄变化，固定间隔不够智能。

#### 2.3.1 智能间隔自适应

| 功能 | 说明 |
|------|------|
| 月龄感知 | 根据宝宝出生日期自动计算月龄，动态推荐间隔 |
| 间隔推荐表 | 内置各月龄的标准喂养间隔（0-3月: 2-3h, 3-6月: 3-4h, 6-12月: 4-5h） |
| 自动调整 | 用户可开启“智能模式”，系统根据近期完成记录自动微调间隔 |
| 渐进过渡 | 间隔调整采用渐进式（每次 ±15 分钟），避免突变 |

#### 2.3.2 智能提醒文案

| 功能 | 说明 |
|------|------|
| 动态文案 | 根据时间/天气/事项类型生成不同文案（“宝宝可能饿了，该喂奶了”） |
| 上次记录 | 推送中带上次完成时间和间隔（“距上次喂奶已 3 小时 20 分”） |
| 鼓励语 | 连续完成率高的用户，推送中附加鼓励语（“本周完成率 90%，做得好！”） |

#### 2.3.3 异常预警

| 功能 | 说明 |
|------|------|
| 超时未完成预警 | P0 事项超时 30 分钟仍未确认，推送加急提醒给家庭其他成员 |
| 连续漏喂检测 | 同一事项连续 2 次被“忽略”，触发关怀提醒 |
| 奶量异常 | 每日总奶量低于月龄最低标准时，提示关注 |
| 生长偏离 | 身高/体重连续 2 次测量落在 3rd 百分位以下，建议就医 |

#### 2.3.4 技术实现

| 能力 | 实现方式 |
|------|---------|
| 月龄计算 | 云函数根据 birthday 实时计算，缓存到 babies 文档 |
| 间隔推荐 | constants.ts 新增 `AGE_INTERVAL_TABLE` 映射表 |
| 异常检测 | cron-scan 扩展，扫描时同时检查异常条件 |
| 文案库 | 云函数端维护文案模板池，随机选择 + 变量替换 |

### 方向四：数据导出与分享

V1.0 数据完全锁在小程序内，家长无法导出，体检时也不方便给医生看。

#### 2.4.1 PDF 育儿周报

| 功能 | 说明 |
|------|------|
| 自动生成 | 每周一生成上周报告，含完成率/趋势/类型分布/异常提醒 |
| 手动生成 | 用户可自选时间范围生成报告 |
| 内容模块 | 概览数据 + 趋势图 + 喂养量汇总 + 睡眠时长 + 里程碑进展 |
| 分享 | 生成 PDF 后可分享到微信聊天或保存到手机 |

#### 2.4.2 体检分享卡

| 功能 | 说明 |
|------|------|
| 一键生成 | 选择宝宝 + 时间范围，生成简洁的体检数据卡 |
| 核心数据 | 近期身高体重 + 百分位 + 喂养方式 + 用药记录 + 过敏食物 |
| 图片分享 | 生成图片格式，方便发给医生 |
| 二维码 | 可选生成小程序码，医生扫码查看更详细数据（只读模式） |

#### 2.4.3 数据备份与恢复

| 功能 | 说明 |
|------|------|
| 导出 JSON | 全量数据导出为 JSON 文件，保存到微信文件/手机 |
| 导出 CSV | 统计数据导出为 CSV，方便 Excel 分析 |
| 数据恢复 | 导入 JSON 文件恢复历史数据（换设备/误删场景） |

#### 2.4.4 技术实现

| 能力 | 实现方式 |
|------|---------|
| PDF 生成 | 云函数使用 `puppeteer` 或 `pdfkit` 生成 PDF，上传云存储返回临时链接 |
| 图片生成 | 前端 Canvas 绘制 → `uni.canvasToTempFilePath` 导出图片 |
| CSV/JSON | 前端组装数据 → `uni.saveFile` 保存到本地 |
| 分享 | `uni.shareAppMessage`（小程序内分享）+ 云存储临时链接（PDF） |

### 方向五：体验优化

#### 2.5.1 深色模式

| 功能 | 说明 |
|------|------|
| 自动切换 | 跟随系统深色模式，支持手动覆盖 |
| 主题变量 | uView UI 原生支持深色模式，补充自定义组件的暗色样式 |
| 时段切换 | 夜间喂奶场景自动切换深色（减少屏幕亮度刺激） |

#### 2.5.2 桌面小组件

| 功能 | 说明 |
|------|------|
| 微信浮窗 | 支持添加到微信浮窗，快速查看今日待办 |
| 服务通知 | 订阅消息优化，支持带操作按钮的富文本通知 |
| 快捷操作 | 通知中直接“完成”或“延迟”，无需打开小程序 |

#### 2.5.3 性能优化

| 优化点 | 方案 |
|--------|------|
| 首屏加载 | 分包加载（首页主包，统计/历史/设置为分包） |
| 数据缓存 | 本地 Storage 缓存最近 7 天数据，减少云函数调用 |
| 图片优化 | 宝宝头像使用云存储缩略图（100x100），懒加载 |
| 云函数冷启动 | 定时预热 cron-scan（间隔 ping），减少首次延迟 |

#### 2.5.4 无障碍

| 功能 | 说明 |
|------|------|
| 字体放大 | 支持系统字体大小设置，小程序内同步缩放 |
| 高对比度 | 可选高对比度模式 |
| 语音播报 | 提醒到达时可选语音播报（“该给宝宝喂奶了”） |

### 3.5 tabBar 调整

V1.0 tabBar 为 4 个页签（首页/统计/记录/我的）。V2.0 考虑：

| 方案 | 说明 |
|------|------|
| 保持 4 个 | 首页新增「成长」入口卡片，跳转到成长记录页 |
| 改为 5 个 | 首页 / 统计 / 成长 / 记录 / 我的（微信限制最多 5 个 tab） |

建议保持 4 个 tab，成长记录通过首页入口进入，避免 tab 过多。

## 四、优先级排序

| 优先级 | 需求 | 理由 |
|--------|------|------|
| P0 | 家庭协作（2.1） | families 表已预留，是 V2.0 核心价值，单用户模式是最大痛点 |
| P0 | 数据迁移脚本 | 家庭协作的前置条件，需安全迁移 V1.0 数据 |
| P1 | 喂养量记录（2.2.2） | 喂养是最高频操作，量化数据价值高，开发量适中 |
| P1 | 智能间隔自适应（2.3.1） | 提升核心体验，技术复杂度可控 |
| P1 | 异常预警（2.3.3） | 基于现有 cron-scan 扩展，增量开发 |
| P2 | 身高体重曲线（2.2.1） | 需引入 WHO 标准数据 + 图表组件，开发量较大 |
| P2 | PDF 育儿周报（2.4.1） | 云函数 PDF 生成有技术挑战，可降级为图片分享 |
| P2 | 体检分享卡（2.4.2） | 图片分享方案简单，二维码只读模式需要权限设计 |
| P2 | 发育里程碑（2.2.3） | 需整理大量月龄数据，内容工作量大 |
| P3 | 深色模式（2.5.1） | 锦上添花，uView 原生支持，工作量小 |
| P3 | 数据备份恢复（2.4.3） | 低频使用，可延后 |
| P3 | 桌面小组件（2.5.2） | 受微信平台限制较大，需跟进微信能力更新 |
| P3 | 性能优化（2.5.3） | 持续迭代，无明确截止 |
| P3 | 无障碍（2.5.4） | 用户量达到一定规模后再考虑 |

## 五、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 数据迁移导致 V1.0 数据丢失 | 高 | 迁移前自动备份全量数据到 `backup_` 集合，迁移脚本支持回滚 |
| 家庭协作权限设计不当 | 中 | 最小权限原则，owner 权限最高，member 只能操作自己创建的事项 |
| 实时同步增加云函数调用成本 | 中 | 使用云数据库 `watch` 直连，减少云函数中转；设置同步频率上限 |
| PDF 生成在云函数中超时 | 中 | 限制 PDF 页数，大数据量分段生成；降级为图片分享 |
| WHO 生长曲线数据版权 | 低 | WHO 公开数据可免费使用，需标注来源 |
| 分包后首屏依赖管理复杂 | 低 | 确保首页只依赖主包，分包按需加载 |

## 六、建议开发节奏

| 阶段 | 周期 | 内容 |
|------|------|------|
| Sprint 1 | 2 周 | 家庭协作全流程（families 激活 + 数据迁移 + 成员管理 + 宝宝共享） |
| Sprint 2 | 2 周 | 喂养量记录 + 智能间隔自适应 + 异常预警 |
| Sprint 3 | 2 周 | 身高体重曲线 + 发育里程碑 |
| Sprint 4 | 1.5 周 | PDF 育儿周报 + 体检分享卡 |
| Sprint 5 | 1 周 | 深色模式 + 性能优化 + 分包 |
| 测试 | 1 周 | 全流程回归 + V1.0 → V2.0 升级测试 |

总计约 9.5 周（2 个月+），可根据实际人力调整。
